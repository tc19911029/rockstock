## 根因詳細分析

### P2 — 盤中掃描慢（最關鍵的系統性問題）

**檔案**: `app/api/scanner/chunk/route.ts` line 93-122

```ts
// chunk/route.ts line 93-122
if (marketOpen) {
  const snapshot = await readIntradaySnapshot(market, todayStr);
  if (snapshot && isSnapshotFresh(snapshot, 120_000)) { // ← 120 秒！
    // 快路徑：只返回候選 symbols，前端做精篩
    return apiOk({ results: coarse, ... });
  }
}
// 沒有快取或快取太舊 → 進昂貴路徑
```

**時間分析**：
- cron 每 2 分鐘（XX:00, XX:02, XX:04...）寫入 snapshot
- 使用者 10:00 掃描 → 上一次 cron 是 09:58（2分鐘前）→ `age = 120s` → `isSnapshotFresh` = **false**（<120s才是true，=120s是false）
- → 快路徑被跳過，直接進昂貴的 `scanSOP`（3800檔全量分析）

**為何使用者回報「卡在10%很久」**：
`backtestStore.runScan` 的 Phase 1 拿到股票清單（`scanProgress: 5→15`），Phase 2 `scanChunk` 開始掃描（`scanProgress: 20`）。快路徑直接在 `/api/scanner/chunk` 內返回，不會有 Phase 2 的漫長等待。但因為快取被判定為不夠新，走了 Phase 2 的 `scanSOP`，而 `TaiwanScanner.scanSOP` 要對 3800 檔做完整技術分析，這就是「卡在10%很久」的原因。

**maxDuration=30 秒**：`scanSOP` 要讀本地磁帶、做均線計算、評估規則，30秒對3800檔可能不夠（Vercel limit 30s）。

### P1 — 日期結果重複

**檔案**: `lib/storage/scanStorage.ts` line 138-272

`listScanDates` 的查詢 path：
```ts
// scanStorage.ts line 150
const blobs = await blobListPrefix(`scans/${market}/${direction}/${m}/`);
// 正則抓 ${date}.json
```

`loadScanSession` 的讀取 path：
```ts
// scanStorage.ts line 286
raw = await blobGet(`scans/${market}/${direction}/${mtfMode}/${date}.json`);
// Fallback: 舊路徑
raw = await blobGet(`scans/${market}/${direction}/${date}.json`);
// Legacy:  raw = await blobGet(`scans/${market}/${date}.json`);
```

**問題**：`listScanDates` 只列出 `scans/{market}/{direction}/{m}/` 下的 blob，但 `saveScanSession`（在 `chunk/route.ts`）寫入的路徑可能不包含 date 尾碼，或者寫入的 session 根本不是以 date 為名的（timestamp 在 id 裡不在路徑裡）。

真正問題在 `saveScanSession` 的命名格式：
```ts
// scanStorage.ts line 107-135
const blobPath = `scans/${session.market}/${dir}/${mtf}/${session.date}.json`;
```

`session.date` 是 `YYYY-MM-DD` 字串。所以路徑是 `scans/TW/long/daily/2026-04-10.json`。`listScanDates` 的正則 `/(\d{4}-\d{2}-\d{2})\.json$/` 會 match 到這個格式。

**但為什麼 4/13 和 4/10 結果相同？**

可能原因：
1. **`chunk/route.ts` 的 `/api/scanner/chunk` 根本沒有 `saveScanSession` 呼叫** — 掃描結果只存在記憶體中，沒有寫入 blob。`saveScanSession` 只在 cron (`scan-tw/route.ts`) 和 `backtestStore` 的 session 儲存（有 `saveSession` endpoint）中被呼叫。所以盤中掃描的結果**從未被寫入 blob**，導致 `listScanDates` 無法找到它，只能找到 cron 寫入的版本（而 cron 沒跑或只有一份）。
2. **`scanCacheKey` 和 `fetchCronDates` 的 `useMultiTimeframe` 不同步** — `fetchCronDates` 用前端的 `useMultiTimeframe` 決定 `mtfParam`，`loadCronSession` 也是。但如果使用者在 `useMultiTimeframe=true` 時掃描，`cacheKey` 包含 `mtf=true`，而 cron 存的是 `mtf=false`（因為 cron 只存 daily version），就會找不到快取。

需要檢查：`chunk/route.ts` 是否有 `saveScanSession`？從程式碼來看，**沒有**。盤中掃描的結果不會寫入 blob，只存在記憶體返回給前端。前端的快取（`scanCache` in backtestStore）才是用來加速的。但如果 `backtestStore` 的 cache 是存在記憶體（Zustand），重新整理頁面後就消失了。

### P3 — 走圖日期跳錯 / 只到 2025/4/30

**檔案**: `lib/datasource/LocalCandleStore.ts`, `store/replayStore.ts`, `app/api/stock/route.ts`

**Step 1: `loadLocalCandlesWithTolerance` 落後於真實時間**

```ts
// LocalCandleStore.ts line 137-141
const gap = businessDaysBetween(data.lastDate, asOfDate);
if (gap <= toleranceDays) {
  return { candles: computeIndicators(data.candles), staleDays: gap };
}
return null; // 太舊
```

`toleranceDays = 5`（`stock/route.ts` line 77）。如果下載 cron 沒跑（可能 Vercel 上沒設定），本地資料真的會落後。

**Step 2: API fallback 到 MultiMarketProvider**

```ts
// stock/route.ts line 164-183
for (const candidate of candidates) {
  const result = await dataProvider.getHistoricalCandles(candidate, period, undefined, interval);
  // candles = 從 blob 讀取，若 blob 也沒有就從網路抓
}
```

問題：`period='2y'`（預設），但如果 blob 快取真的只有到 2025-04-30，就只會回那些資料。

**Step 3: `targetDate` 定位**

```ts
// replayStore.ts line 196-217
if (targetDate) {
  const dateIdx = allCandles.findIndex(c => c.date === targetDate);
  if (dateIdx !== -1) { index = dateIdx; }
  else {
    let closest = -1;
    for (let i = allCandles.length - 1; i >= 0; i--) {
      if (allCandles[i].date <= targetDate) { closest = i; break; }
    }
    if (closest === -1) {
      const lastDate = allCandles[allCandles.length - 1]?.date;
      if (lastDate && lastDate < targetDate) {
        const daysGap = (new Date(targetDate).getTime() - new Date(lastDate).getTime()) / 86400000;
        if (daysGap > 30) {
          throw new Error(`K線資料僅到 ${lastDate}...`);
        }
      }
      closest = 0; // ← 這裡：差距 <= 30 天時，用最舊的 K 棒
    }
    index = closest;
  }
}
```

如果 `allCandles = ['2024-01-01', ..., '2025-04-30']`（2025年的資料，沒到2026），而 `targetDate = '2026-04-10'`：
- `findIndex` → -1
- 迴圈找不到（所有 date 都 > targetDate? 不對，'2025-04-30' < '2026-04-10'）
- 實際上：'2025-04-30' < '2026-04-10' → 會找到 `closest = last index`
- 但 `lastDate (2025-04-30) < targetDate (2026-04-10)` → gap = 超過 30 天
- → 會 throw 錯誤

所以 `targetDate` 為什麼沒有被拒絕？**可能是 `targetDate` 本身格式錯誤**。如果傳進來的是 `'2025-04-10'`（不是 '2026-04-10'），而資料範圍是 `['2025-01-01', ..., '2025-04-30']`，那 `findIndex('2025-04-10')` 會找到精確匹配（如果有的話），或者找到倒數第二個。

真正問題在 **`ScanChartPanel` 呼叫 `loadStock` 時帶的 `scanDate`**：
```ts
// ScanChartPanel.tsx line 74
loadStock(selectedStock.symbol, '1d', '2y', scanDate)
// scanDate 來自 backtestStore.getState().scanDate
```

而 `backtestStore.scanDate` 是在 `loadCronSession` 或 `runScan` 時設定的。如果 `loadCronSession` 因為找不到 blob 而沒設定，`scanDate` 就保持舊值。

### P5 — 陸股快取失效

**檔案**: `lib/datasource/IntradayCache.ts` line 206-217

```ts
// EastMoney 即時報價
quotes.push({
  symbol: q.code, name: q.name,
  open: q.open, high: q.high, low: q.low, close: q.close, volume: q.volume,
  prevClose: 0,     // ← 問題根源
  changePercent: 0, // ← 問題根源
});
```

`chunk/route.ts` 的粗篩：
```ts
// chunk/route.ts line 107-109
const gapPct = Math.abs(q.open - q.prevClose) / q.prevClose * 100; // 除以 0 → NaN
const isGap = gapPct >= 2; // NaN >= 2 → false
const isSurge = Math.abs(q.changePercent) >= 5; // 0 >= 5 → false
return isGap && isSurge; // → false
```

**所有陸股在盤中都過不了粗篩，導致 `coarse` array 為空。** 然後整個快路徑返回 `results: []`，或進入昂貴路徑。

此外，因為 `changePercent=0`，`CoarseScanner` 的條件4（`q.changePercent > 0`）永遠失敗，導致陸股必須靠其他4個條件過關，篩選標準變得更嚴格。

### P6 — 打板 UI 只看到 7 檔

**檔案**: `features/scan/components/DabanResultsTable.tsx` line 334

```tsx
<div className="overflow-x-auto max-h-[480px] overflow-y-auto">
  <table className="w-full text-sm">
```

macOS Safari/Chrome 的已知行為：當一個父容器同時有 `overflow-x: auto` 和 `overflow-y: auto`（或 `max-height` + `overflow-y: auto`），瀏覽器在計算 `max-height` 的可用空間時，**如果存在水平 scrollbar，會把這個 scrollbar 的高度（約17px）從垂直可用空間扣除**。同時，沒有 `table-layout: fixed` 會讓瀏覽器根據內容寬度計算 table 寬度，某些長文字欄位（如中文名稱、產業分類）會把格子撐寬，觸發水平 scrollbar，進而吃掉垂直空間。

## 修復方案

### P2 Fix: 調整快取 threshold + 增加 cron timeout

**`app/api/scanner/chunk/route.ts`**：


**`app/api/cron/update-intraday/route.ts`**：
```ts

```

### P5 Fix: 補足陸股 prevClose

在 `IntradayCache.refreshIntradaySnapshot()` 的 EastMoney 分支，加入 MA base 補足：
```ts
// 讀 MA base 補足 prevClose
const maBase = await readMABase(market, today);
for (const q of quotes) {
  const maEntry = maBase?.data[q.symbol];
  if (q.prevClose === 0 && maEntry?.closes?.length > 0) {
    q.prevClose = maEntry.closes[maEntry.closes.length - 1]; // 昨天的收盤價
    q.changePercent = q.prevClose > 0
      ? Math.round((q.close - q.prevClose) / q.prevClose * 10000) / 100
      : 0;
  }
}
```

### P3 Fix: targetDate 定位改善

**`store/replayStore.ts`**：
- 當 gap > 30 天，throw 明確錯誤，讓 UI 顯示有意義的訊息
- 當 gap <= 30 天但找不到精確 match，不應設 `index=0`（這是從最舊開始），應該 throw 或 warn

**`app/api/stock/route.ts`**：
- 當 `local=1` 回傳的 `staleDays > 0`，在 API 回應中加入 `staleDays` 欄位
- 前端收到 `staleDays > 0` 時顯示警告：「本地資料落後 N 天，可能影響定位」

### P1 Fix: 確認掃描結果有被快取

在 `backtestStore.runScan` 的最後，加入明確的快取寫入：
- 掃描完成後，立即呼叫 `saveScanSession` 寫入 blob（不只在 cron）
