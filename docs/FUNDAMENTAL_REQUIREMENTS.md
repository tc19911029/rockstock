## 一、資料分層架構（4 層，互不污染）

### Layer 1：歷史日K主資料庫

- 每個交易日收盤後，必須更新全台股與全陸股日K資料
- 歷史日K資料必須獨立存放
- **歷史資料一旦封存（sealed），不得被盤中資料覆蓋**
- 所有歷史掃描、回測、開盤前查看，都只能讀取封存後的歷史資料
- 不可只更新局部標的，必須覆蓋全市場
- 歷史日K資料可持續累積

### Layer 2：盤中即時快取層

- 盤中資料必須獨立於歷史資料存在
- 盤中只更新當日未完成K線與必要即時欄位
- **盤中資料不得直接覆蓋歷史正式紀錄**
- 全市場快照存為單一檔案（非逐檔存取），確保讀取效率
- 台股開盤時，持續更新台股盤中快照
- 陸股開盤時，持續更新陸股盤中快照

### Layer 3：個股高頻走圖層

- 即時走圖資料流必須獨立於全市場掃描資料流
- 使用者正在查看的股票與持倉股票，享有高於全市場掃描的更新優先級
- 走圖更新不可拖慢或干擾全市場掃描
- 最多同時追蹤 10-20 檔高頻更新

### Layer 4：掃描結果層

- 每次掃描結果都必須用複合主鍵唯一標識：
  `market + strategy + trade_date + session_type + scan_timestamp`
- **不得以 latest 結果直接覆蓋不同交易日掃描紀錄**
- 盤中掃描結果只能作為 intraday snapshot 保存
- 正式歷史掃描結果只能在收盤後生成與封存
- session_type 區分：pre_open / intraday / close / post_close

## 二、掃描架構（兩級掃描）

### 第一級：全市場粗掃（Coarse Scan）

- 使用 Layer 2 的全市場快照進行，**不逐檔讀取歷史 K 線**
- 粗篩條件只使用快照可取得的欄位（價格、漲跌幅、量能、均線相對位置）
- 目標：將全市場（1900 TW / 4000 CN）壓縮到幾十檔候選
- 效能要求：< 3 秒（含手機網路環境）

### 第二級：候選池精掃（Fine Scan）

- 只對粗掃通過的候選股票，讀取完整歷史 K 線
- 跑完整六條件 + 戒律 + 淘汰法 + MTF filter
- 候選池精掃才允許使用更高頻資料
- 效能要求：< 30 秒

### 禁止事項

- **不得以逐檔高頻 minute K 的方式直接掃描全市場**
- 不得每按一次掃描就重新向 API 把全市場每一檔完整K線抓一遍

## 三、API 使用原則

### 分場景分工

| 場景 | TW Provider | CN Provider |
|------|------------|------------|
| 收盤後日K (Layer 1) | EODHD(主)→FinMind(備) | EODHD(主)→EastMoney(備) |
| 盤中全市場快照 (Layer 2) | TWSE mis 批量 | EastMoney push2 批量 |
| 走圖分鐘K (Layer 3) | Fugle | EastMoney klt |
| 歷史掃描 | 本地/Blob（零 API） | 本地/Blob（零 API） |
| 盤中粗掃 | Layer 2 快照（零額外 API） | Layer 2 快照（零額外 API） |
| 盤中精掃 | Blob candles + Layer 2 | Blob candles + Layer 2 |

### 分優先級節流

| 優先級 | 任務 |
|--------|------|
| 1（最高） | 使用者正在看的走圖 |
| 2 | 持倉股票即時監控 |
| 3 | 候選池精掃 |
| 4 | 全市場粗掃 |
| 5（最低） | 非必要背景刷新 |

### 硬性規則

- 收盤後全市場日K更新、盤中全市場粗掃、個股高頻走圖，必須分為不同資料鏈路
- 各任務需有獨立節流、重試、快取與優先級策略
- 系統需自行控制 API 使用量，避免超過供應商限制
- API 取得方式由系統底層設計，不可每次因新需求調整就被重寫

## 四、盤中掃描是核心能力

- 盤中掃描不是附加功能，是系統核心能力之一
- 系統要能支援台股盤中掃描
- 系統要能支援陸股盤中掃描
- 掃描必須建立在最新盤中資料之上
- 掃描必須可以反覆執行

### 台股盤中時程

| 時段 | 行為 |
|------|------|
| 08:30-08:59 | 只讀歷史封存資料，不做盤中掃描 |
| 09:00-12:59 | 全市場粗掃中頻 + 候選池高頻 + 持倉/走圖最高頻 |
| 13:00-13:30 | 提高全市場粗掃頻率 |
| 收盤後 | 封存正式日K + 生成正式掃描結果 |

### 陸股盤中時程

| 時段 | 行為 |
|------|------|
| 開盤前 | 只讀歷史封存資料 |
| 09:30-14:29 | 全市場粗掃中頻 + 候選池高頻 |
| 14:30-15:00 | 提高全市場粗掃頻率 |
| 收盤後 | 封存正式日K + 生成正式掃描結果 |

## 五、選股架構原則

### 選股層（決定選不選）= 純書本規則

1. 六條件前 5 個 = 核心門檻（isCoreReady），第 6 個 KD/MACD = 候補加分
2. 短線第 9 條（KD 向下不買）、第 10 條（上影線 > 1/2 不買）
3. 10 大戒律（朱老師 p.54）

- 共振訊號數（BUY/ADD 數量 + 跨群組共振）
- 高勝率進場加分（寶典 Part 12）

### 禁止事項

- 任何新增的篩選條件必須確認出自哪本書的哪個章節，不能自創
- 排序因子限於朱/林體系內
- 不加非書本的因子參與選股

## 六、資料鏈路完整性

### 收盤後鏈路

```
收盤後取得全市場最新日K資料
→ 更新歷史主資料庫（Layer 1）
→ 標記 sealed
→ 校驗缺漏與補抓
→ 生成正式掃描結果（Layer 4, post_close）
→ 更新 MA base 快取（供隔日盤中粗掃用）
→ 清理當日 intraday cache
```

### 盤中鏈路

```
開盤中取得全市場即時快照（Layer 2, 批量 API）
→ 控制更新頻率（cron 每 2 分鐘）
→ 用戶按掃描 → 讀快照 → 粗掃
→ 粗掃結果進候選池 → 精掃（讀 Blob 歷史 + 快照）
→ 輸出掃描結果（Layer 4, intraday snapshot）
→ 走圖/持倉獨立高頻通道（Layer 3）
```

## 七、趨勢判定（朱家泓寶典 p.21-35）

### 短線轉折波（頭/底）

- 以**收盤價 vs MA5** 分段：close > MA5 = 正價區，close < MA5 = 負價區
- 「漲過/跌破 MA5」按**收盤價**判定，盤中穿越不算
- 正→負（跌破日）：取正段 K 棒 + 跌破當天，max(high) = **頭**
- 負→正（突破日）：取負段 K 棒 + 突破當天，min(low) = **底**
- 交界日**雙重計算**：既是舊段結束候選，也是新段起點（close 在新段那一側）
- 頭底**嚴格交替**，不會連續兩個頭或兩個底
- **Provisional pivot（進行中段）**：
  - 當前進行中的正價段，其 running max(high) 視為 provisional 頭
  - 當前進行中的負價段，其 running min(low) 視為 provisional 底
  - **僅用於即時趨勢判定**（`detectTrend`）：讓「價格已漲過前頭／跌破前底」立刻反映為 頭頭高／底底低
  - **不用於走圖頭底標記、切線連線、選股條件、警示訊號**——這些用已確認 pivot（`includeOpen=false`）
  - 理由：provisional 還沒 MA5 反向穿越確認，不是真正的「頭/底」；拿來畫切線會用到未完成的錨點；但對趨勢方向判定已足夠

### 趨勢定義（本系統不可覆寫的判定規則）

以最近兩個頭與最近兩個底的相對高低：

| 頭 | 底 | 趨勢 |
|---|---|---|
| 頭頭高 | 底底高 | **多頭** |
| 頭頭低 | 底底低 | **空頭** |
| 其他組合（含頭高底低、頭低底高、波浪不完整） | | **盤整** |

**實作位置**：[lib/analysis/trendAnalysis.ts](../lib/analysis/trendAnalysis.ts) `detectTrend()` 與 `findPivots()`

**不可改動：**
- 不可加振幅門檻、最小段長度、zigzag 過濾等「清理」邏輯
- 不可用 close-vs-MA5 以外的分段方式（例如 pivot-to-pivot leg 視窗）
- 不可 hard-code 這條趨勢規則於其他檔案（UI、store、回測腳本）—— 必須從 `trendAnalysis.ts` 讀

## 九、本文件的目標

避免系統因反覆修改導致：
- 收盤資料不完整
- 盤中資料不更新
- API 超限
- 掃描失效
- 台股或陸股其中一邊不能用
- 走圖與掃描對不上
- 歷史資料與即時資料斷裂
- 手機（Vercel）掃描超時失敗
- 不同日期掃描結果互相覆蓋
